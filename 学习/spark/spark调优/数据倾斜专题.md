## 数据倾斜的排查
### 1.数据倾斜的现象
一个job/stage 中的大部分task执行的很快，但有极个别的task执行的非常慢，任务整体执行进度在99%，就跟卡住了一样，比如1000个task，99%都在5-30分钟执行完了，有2-3个2个小时还没执行完，甚至任务失败。
### 2.为啥会有数据倾斜
数据倾斜的本质是shuffle的时候分布不均匀
一般来说，map端切分数据的时候会很均匀，所以map端基本不会产生数据倾斜。
大部分都是reduce端产生数据倾斜，reduce端的数据是shuffle过程划分来的，shuffle机制是按照key进行哈希取模，或者范围分区来决定数据去向的，如果某个key值的数据量远远大于其他key值，那么承载这些key值的reduce会收到海量数据，这个reduce会成为性能瓶颈。

### 3.数据倾斜的危害

大数据是一个分布式系统，比如有10台机器，我们跑任务的时候肯定是想把任务均分给10台机器，数据分布均匀，每个节点的计算量和数据量大致相当，所有任务几乎同时完成，总时间最短。但是实际情况不可能这么美好，肯定有机器跑的快，有机器跑的慢，这时候跑的慢的就成为性能瓶颈了，其他已经跑完的机器一直在等待。数据倾斜会导致少数节点过载、长时间运行，而大部分节点早已空闲，集群资源利用率极低，储资浪费了宝贵的计算资源。还有就是因为数据倾斜，会导致任务整体执行时间过长，数据时效性得不到保障，严重的数据倾斜会造成任务失败，数据重复或者确实，进而影响数据质量。从集群层面，数据倾斜shuffle过程中，reduce会从大量 Map 拉取数据，导致该节点网络、磁盘、CPU 负载极高，可能成为集群的故障点。

1. **木桶效应导致任务执行时间过长**：整个作业的完成时间取决于最慢的那个 Reduce 任务。倾斜使得某个 Reducer 处理的数据量远超其他 Reducer，成为瓶颈，其他 Reducer 早早完成后闲置等待，拖慢整体进度。
    
2. **资源利用率低下**：大量集群节点处于空闲或低负载状态，而少数热点节点资源被耗尽（CPU 飙升、内存溢出、网络拥堵），无法充分利用分布式集群的并行能力。
    
3. **稳定性风险**：热点节点可能因内存不足导致 OOM，或磁盘 I/O 饱和、网络超时，引起任务失败。为了容错，任务可能被重试，进一步加剧资源消耗，甚至导致整个作业无法成功。
    
4. **数据时效性受损**：ETL 任务无法在规定时间窗口内完成，影响下游数据依赖，可能导致报表延迟、实时指标失准。

### 4.从任务角度看待数据倾斜的分类

1. **数据读取倾斜**，也就是map端倾斜，比如某个map，或者task读取数据阶段长时间无法完成，这通常是文件分块过大，或者这个数据块有异常。还有就是这个节点的性能太差，网络太慢了。map端倾斜的情况很少
2. **计算过程中数据倾斜**，比如 group by，join（不包括mapjoin，广播join），order by，窗口函数等场景时候，同一个key，通常是一个或者多个字段或者表达式的组合的处理时间过长。这个工作中遇到的比较多，情况也比较复杂。其他某些操作只能由单一节点进行。一切需要维护一个全局状态的大多数操作，如order by排序、Limit、count distinct，等，一般会安排到一个节点来执行。
3. **写数据倾斜：** 数据生成在临时文件夹后，由于数量巨大，重命名和移动的操作非常耗时。这通常发生在动态分区导致小文件的情况。

### 3.join系列数据倾斜

#### 1.大表遇到小表
背景：当遇到一个大表和一个小表进行join操作时，大表数据分布不均匀时产生倾斜。
**hive任务**  我们在map端进行了join操作，省去了reduce阶段，即没有shuffle过程，没有reduce任务，避免了数据倾斜。

**广播关联小表**（通过参数 spark.sql.autoBroadcastJoinThreshold 控制，官方默认值10m，设置-1表示禁用广播，我们设置100M）的话会采用广播关联，即将小表的全部数据传输到各节点的内存中，通过直接的内存操作快速完成关联。这种方式最大的好处是避免了对主表的数据进行 shuffle，但会增加任务使用的内存量。另外特别说明 3 点：
- 目前 sparksql 优化器尚不能非常准确地判断一个子查询结果（也被当成一张小表）是否适合进行广播；
- 关于表广播的参考hive的mapjoin使用测试，比如left join不能广播左表，full join不生效等。
- 某些情况下会有类似：Kryo serialization failed: Buffer overflow 这样的 OOM 出现，并 “To avoid this, increase spark.kryoserializer.buffer.max value”。但其实这样设置会无效。实质原因是：虽然某张表小于 32M，但由于高度压缩后，解压结果的行数达到了数千万，造成了节点的 OOM。这个时候，只能手动禁掉广播关联。

#### 2.大表join 大表 
##### 1. **倾斜 Key 单独处理（Salt + Union All）** —— 最通用、最可控

**核心思路**：把热点 Key 和非热点 Key 分开 Join，最后合并。对热点 Key 加上随机前缀打散，对小表侧对应 Key 做膨胀（复制多份）后再 Join。

##### 2.拆表拆热点key值：

1. 基于主表统计获得Top热点的属性值；用热点属性值将被关联维表拆成热点小表和非热点表，同时也将主表拆成热点主表和非热点主表；
2. 热点小表通过Mapjoin与热点主表join，非热点表与非热点主表join
3. 最终两部分再Union到一起，完成数据关联。

##### 3. **开启 Skew Join 优化（Hive）** —— 自动处理倾斜
**原理**：运行时检测倾斜 key，自动启动两个作业：第一个作业将倾斜 key 用随机前缀打散后 Join，第二个作业处理非倾斜 key。但对大表 Join 大表场景，效果有限（因为需要额外扫描一次数据）。


##### 4. **使用 Bucket Map Join 或 SMB Join** —— 预分桶避免 Shuffle

如果两个大表事先按 Join 字段分桶（且桶数成倍数关系），并且数据排序，可以使用 Sort Merge Bucket (SMB) Join，避免全量 Shuffle。
**条件**：两表分桶字段、桶数（倍数）、排序完全一致。适合长期优化。

##### 5. **过滤空值或无效 Key**

很多倾斜由空值引起。如果业务允许，在 Join 前过滤掉空值：
或者将空值随机打散：`CASE WHEN key IS NULL THEN concat('null', rand()) ELSE key END`。

##### 6. **扩大 Reduce 数量 + 增加资源**

- 增加 Reducer 数（`set mapreduce.job.reduces = 500;`）虽然不能解决单个 Key 倾斜，但能分散非热点 Key，为倾斜 Key 留出更多内存。
    
- 给倾斜 Reducer 增加内存（`mapreduce.reduce.memory.mb`），避免 OOM。


#### group by 产生的数据倾斜

##### 1. 采用两阶段聚合的方式
两阶段聚合指的是先局部聚合再全局聚合。局部聚合时候给每个key值加上随机前缀进行打散，原本相同的key值会变成不同的新key值，便可以让原来由一个task处理的数据根据加上随机前缀后的新key值分散到多个Task上做聚合，从而缓解单个task处理数据量过多的问题。再去除随机前缀做全局聚合，既可以得到最终结果。

 /// map端聚合 在 Map 阶段对同一个 Map Task 内的数据先做一次**局部聚合**（类似 Combiner），减少 Shuffle 到 Reduce 的数据量。
#### 2. 使用自带的倾斜优化参数

**是否优化倾斜的查询为两道作业。其实现方法是在group by时启动两个MR job。第一个job会将map端数据随机输入reducer，每个reducer做部分聚合，相同的key就会分布在不同的reducer中。第二个job再将前面预处理过的数据按key聚合并输出结果，这样就起到了均衡的效果。**
**第一阶段**：
- Map 端输出时，为 key 加随机前缀（比如 `key + "_" + random(1..N)`）。
- 随机打散后的 key 进入 Reducer，每个 Reducer 做局部聚合（部分聚合）。
- 原本倾斜的 key 现在被分散到多个 Reducer，每个 Reducer 只处理该 key 的一部分数据。
**第二阶段**：
- 去掉随机前缀，对第一阶段的局部聚合结果按真实 key 做最终聚合。
- 此时每个 key 只出现在一个 Reducer 上，但由于数据量已经大大减少（经过一次聚合），压力显著降低。
