### 1.查看任务异常的日志

#### 先查看调度日志
如果能看到问题，直接解决，比如一些简单的报错，上游字段变更，字段不存在，插入表字段个数不一致。
#### 如果调度看不出问题，就从日志里找yarn的url或者找到对应的app_id去yarn上查找

### 2.看yarn日志排查

#### hive任务

##### 1.查看报错原因：

直接去看app_id/job报错的原因，点击到具体的task里面去查看报错，看看是什么原因，当然也有可能是appmaster报错，不是task报错，那么去appmaster报错查看就行

##### 2.定位报错SQL：
如果代码比较长，我们可以通过hive看执行计划，通过执行计划中关键字的一些输入输出表，operator算子，stage名等同时结合日志定位哪一段SQL报错

##### 3.解决报错
针对这段sql的报错具体问题具体解决，比如数据倾斜拆分sql，参数优化sql优化等等


**注意hive引擎的执行，会跟shuffle算子等把同一个sql分成不同的stage执行，每个stage都是独立的application运行的,这点跟spark有区别的。**




##### Spark任务
##### 1.查看报错原因：
spark 任务首先点进yarn 日志，定位哪个是job，哪个是stage 异常，然后找到对应的stage里的task日志看报错原因。当然也有可能是driver的失败报错，直接看driver的日志即可。
##### 2.定位报错SQL：
如果说代码sql比较复杂，那么我们可以先通过spark的webui的执行计划，通过执行计划中的关键字一些输入输出表，operator算子，字段名，等同时结合日志定位那一段sql报错
##### 3.解决报错
针对这段sql的报错具体问题具体解决，比如数据倾斜拆分sql，参数优化sql优化等等

### **spark任务如何异常定位具体哪段SQL排查？**

| 概念              | 划分依据              | 说明                                                                                                                      |
| --------------- | ----------------- | ----------------------------------------------------------------------------------------------------------------------- |
| **Application** | **一段完整的SQL**      | 基于 Spark 构建的用户程序。我们提交的一次作业其实就是一个应用                                                                                      |
| **Executor**    | **container**     | **为工作节点上的应用程序启动的进程，它运行任务并将数据保存在内存或磁盘存储中**                                                                               |
| **Job**         | **Action**        | 一个Action产生一个Job。Job是Spark作业调度的最大单位。                                                                                     |
| **Stage**       | **Shuffle (宽依赖)** | 一个Job根据宽依赖被切分为多个Stage。Stage是任务执行（Task调度）的单位。<br><br>每个作业都被分成更小的任务集，称为stage阶段，这些阶段相互依赖（类似于 MapReduce 中的 map 和 reduce 阶段） |
| **Task**        | **分区数**           | 一个Stage会根据最终RDD的分区数，生成多个**Task**。每个Task处理一个分区的数据。Task是最终在Executor上执行的工作单元。                                              |
**执行流程简化视图：** **一条SQL → 遇到Action → 生成一个Job → Job根据宽依赖划分成多个Stage → 每个Stage根据分区数生成多个Task → Task被调度到Executor上并行执行。**
